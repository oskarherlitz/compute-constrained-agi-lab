# Build logs

Weekly public updates on what shipped, what changed, and what was learned.

## Format

Each log should cover:

1. **What shipped** — code, charts, data, or writing
2. **What assumption changed** — stronger, weaker, or new
3. **One scenario or chart** — make the constraint concrete
4. **Next action** — one clear priority for the following week

## Naming

`build-log-01.md`, `build-log-02.md`, etc.

## Target

At least 4 substantive build logs over the summer.
